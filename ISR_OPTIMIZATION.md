# ISR Оптимизация для экономии Build минут на Vercel

## Что такое ISR?

**ISR (Incremental Static Regeneration)** - это технология Next.js, которая позволяет:
- Генерировать только важные страницы при build
- Остальные страницы создаются по требованию (on-demand)
- Автоматически обновлять страницы по расписанию

## Проблема без ISR

### Текущая ситуация (до оптимизации)

**Сателлит с 1000 страниц:**
```
generateStaticParams() возвращает ВСЕ 1000 страниц
↓
Next.js генерирует все 1000 страниц при build
↓
Build время: 2-3 минуты
```

**20 сателлитов в день:**
```
20 сателлитов × 3 минуты = 60 минут/день
60 минут × 30 дней = 1,800 минут/месяц
```

**Проблема:** Vercel Pro дает только 6,000 минут/месяц
- 1,800 минут на генерацию
- Остается 4,200 минут на другие проекты
- Нет запаса на ошибки/пересборки

## Решение с ISR

### Новая стратегия

**Сателлит с 1000 страниц:**
```
generateStaticParams() возвращает только ТОП-50 страниц
↓
Next.js генерирует только 50 страниц при build
↓
Build время: 10-30 секунд
↓
Остальные 950 страниц генерируются по требованию (первый визит)
↓
Кешируются и обновляются по revalidate
```

**20 сателлитов в день:**
```
20 сателлитов × 0.5 минуты = 10 минут/день
10 минут × 30 дней = 300 минут/месяц
```

**Экономия:** 1,800 - 300 = **1,500 минут/месяц (83% экономия!)**

## Реализация в проекте

### 1. Long-tail страницы (самые многочисленные)

**Было:**
```javascript
export async function generateStaticParams() {
  return longTailPages.map(page => ({ slug: page.slug }));
  // Генерирует ВСЕ 400-600 страниц при build
}
```

**Стало:**
```javascript
export async function generateStaticParams() {
  const topPages = longTailPages
    .sort((a, b) => (b.priority || 0) - (a.priority || 0))
    .slice(0, 50);  // Только ТОП-50
  
  return topPages.map(page => ({ slug: page.slug }));
}

export const revalidate = 3600; // Обновление каждый час
```

**Результат:**
- Build: 50 страниц вместо 600
- Экономия: ~90% build времени
- Остальные 550 страниц: генерируются по требованию

### 2. Город + Направление страницы

**Было:**
```javascript
export async function generateStaticParams() {
  // 120 городов × 12 направлений = 1,440 страниц
  cities.forEach(city => {
    directions.forEach(dir => {
      params.push({ city: city.slug, direction: dir.id });
    });
  });
}
```

**Стало:**
```javascript
export async function generateStaticParams() {
  // Только ТОП-5 городов × ТОП-10 направлений = 50 страниц
  const topCities = cities.slice(0, 5);
  const topDirections = directions.slice(0, 10);
  
  topCities.forEach(city => {
    topDirections.forEach(dir => {
      params.push({ city: city.slug, direction: dir.id });
    });
  });
}

export const revalidate = 21600; // Обновление каждые 6 часов
```

**Результат:**
- Build: 50 страниц вместо 1,440
- Экономия: ~97% build времени
- Остальные 1,390 страниц: генерируются по требованию

### 3. Статьи и сравнения

**Было:**
```javascript
// Все статьи
export async function generateStaticParams() {
  return howToPages.map(p => ({ slug: p.slug })); // 30-50 страниц
}

// Все сравнения
export async function generateStaticParams() {
  return comparisonPairs.map(p => ({ slug: p.slug })); // 100+ страниц
}
```

**Стало:**
```javascript
// Только ТОП-20 статей
export async function generateStaticParams() {
  return howToPages.slice(0, 20).map(p => ({ slug: p.slug }));
}
export const revalidate = 21600;

// Только ТОП-30 сравнений
export async function generateStaticParams() {
  return comparisonPairs.slice(0, 30).map(p => ({ slug: p.slug }));
}
export const revalidate = 43200;
```

## Как работает ISR на практике

### Сценарий 1: Популярная страница (в топ-50)

```
1. Build time: страница генерируется статически
2. Первый визит: мгновенная загрузка (уже готова)
3. Через 6 часов: автоматическое обновление в фоне
4. Следующие визиты: обновленная версия
```

### Сценарий 2: Непопулярная страница (не в топ-50)

```
1. Build time: страница НЕ генерируется
2. Первый визит: 
   - Next.js генерирует страницу on-demand (~200-500ms)
   - Кеширует результат
   - Отдает пользователю
3. Второй визит: мгновенная загрузка (из кеша)
4. Через 6 часов: автоматическое обновление в фоне
5. Следующие визиты: обновленная версия
```

### Сценарий 3: Страница вообще не существует

```
1. Визит на /longtail/несуществующая-страница
2. Next.js пытается сгенерировать
3. notFound() возвращает 404
4. Пользователь видит страницу 404
```

## Стратегия revalidate

### Разные интервалы для разных типов страниц

```javascript
// Long-tail: часто обновляются (новый контент)
export const revalidate = 3600; // 1 час

// Направления: средняя частота обновлений
export const revalidate = 21600; // 6 часов

// Аудитории: редко меняются
export const revalidate = 43200; // 12 часов

// Периоды: очень редко меняются
export const revalidate = 86400; // 24 часа
```

### Почему разные интервалы?

1. **Long-tail (1 час):**
   - Новый контент добавляется часто
   - Важна свежесть для SEO
   - Много страниц, но кеш эффективен

2. **Направления (6 часов):**
   - Контент стабильный
   - Обновления редкие
   - Баланс между свежестью и нагрузкой

3. **Аудитории (12 часов):**
   - Очень стабильный контент
   - Обновления крайне редкие
   - Минимальная нагрузка

4. **Периоды (24 часа):**
   - Исторические данные
   - Практически не меняются
   - Максимальная эффективность кеша

## On-Demand Revalidation

### Дополнительная оптимизация

Можно добавить API endpoint для принудительного обновления:

```javascript
// src/app/api/revalidate/route.js (уже существует)
export async function POST(request) {
  const { path, secret } = await request.json();
  
  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }
  
  await revalidatePath(path);
  return Response.json({ revalidated: true });
}
```

**Использование:**
```bash
curl -X POST https://satellite.vercel.app/api/revalidate \
  -H "Content-Type: application/json" \
  -d '{"path": "/longtail/some-page", "secret": "SECRET"}'
```

## Преимущества ISR для SEO

### 1. Быстрая индексация

- Топ-50 страниц доступны сразу после деплоя
- Поисковики индексируют их мгновенно
- Остальные страницы индексируются по мере обхода

### 2. Свежий контент

- Автоматическое обновление по расписанию
- Не нужно пересобирать весь сайт
- Контент всегда актуальный

### 3. Масштабируемость

- Можно иметь миллионы страниц
- Build время не зависит от общего количества
- Генерация только по требованию

### 4. Производительность

- Все страницы кешируются на CDN
- Мгновенная загрузка для пользователей
- Отличные Core Web Vitals

## Рекомендации по приоритизации

### Какие страницы включать в топ-50?

1. **Главная страница** - всегда
2. **Топ-10 направлений** - самые популярные ниши
3. **Топ-5 городов** - Москва, СПб, Екатеринбург, Казань, Новосибирск
4. **Топ-20 long-tail** - по трафику из аналитики
5. **Топ-10 статей** - самые читаемые

### Как определить приоритет?

```javascript
// Добавить поле priority в данные
const longTailPages = [
  {
    slug: 'ai-copywriting-dlya-nachinayushchih',
    title: '...',
    priority: 100, // Высокий приоритет
  },
  {
    slug: 'redkiy-zapros',
    title: '...',
    priority: 1, // Низкий приоритет
  },
];

// Сортировать по priority
const topPages = longTailPages
  .sort((a, b) => (b.priority || 0) - (a.priority || 0))
  .slice(0, 50);
```
